# /podcast morning — per-show editorial

This file is loaded by `skills/podcast/SKILL.md` when the user invokes
`/podcast morning`. It contains the per-step editorial instructions for the
persona subagents. It is **editorial prose, not orchestration logic**;
orchestration lives in `SKILL.md`.

## Show identity

- **Genre:** 每日综合播客 (morning show).
- **Cadence:** daily, morning slot.
- **Length:** 目标约 7000 字（~20 分钟）；硬下限 6500 非空白字（≈18 分钟，产品下限），低于触发长度门 `check_min_chars` 扩写重跑.
- **Structure:** 四段事件中心随笔 (4-段事件中心结构；可证伪观点织入正文，不单列「我下注」格式段).
- **Voice:** 卞旸, 第一人称, 自然口语, 有观点态度, 不报菜名式陈列.

## Per-step editorial (per the Adam morning template)

### Collection (达芬奇)

在五域（tech / market / science / geo / culture）中按量级/显著性挑 3-5
个带 `domain` 标签的候选 topic_tag；从 `vault.subjective_dir` 抽一条
主观笔记作 `pkos_note`（`{id, title, excerpt}`）。然后用
`${CLAUDE_PLUGIN_ROOT}/skills/podcast-studio-prep/scripts/orchestrator.py check` 跑 3 次
（每条只改 `--force-domain` 与 `--force-contrarian`），每次加
`--show-type morning` 与 `--required-domains
"tech,market,science,geo,culture"` 让 prep 在代码侧强制跨域配额选源：

- 路 A: `--force-domain philosophy --force-contrarian lesswrong`
- 路 B: `--force-domain cognition --force-contrarian marginal-revolution`
- 路 C: `--force-domain history --force-contrarian stratechery`

3 条命令返回的完整 JSON brief 整段原样贴进素材摘要，分别放进 3 个
` ```json ` 代码块，代码块前用一行写明 `brief-A` / `brief-B` / `brief-C`。
3 份 brief 一个字段都不能漏、不能改写。

### Drafts A/B/C (达芬奇) — 4 段事件中心随笔骨架

每路从素材摘要里取对应 `brief-X`，基于它写成播客文字稿（约 7000 字），
以卞旸第一人称写作，Persona=卞旸。每段是独立段落级章节，
段落之间用承接句（"顺着这条线往下看"/"另一件值得说的事是"/"如果把这
事放到更大的坐标系里"）连接：

**叙事/情绪弧（Phase 5, 4 段结构之上的纵向情绪曲线）**：

播客不是「概览 + 知识点」陈列——它是一条情绪曲线，听众从开场的
冷开场被拉进来，经历一个赌注被抬高的过程，最后在收尾的落点
带着一个还没放下的问题离开。每段在 4 段结构之上还要承担弧
上的具体情绪职责：

- **冷开场 (cold open)** —— ① 之前 30-60 字钩子：一句具体的
  画面/反直觉观察/我最近一直放不下的小事；不要从"各位听众朋
  友"或"今天我们来聊聊"起手。冷开场要直接把人扔进一个画面，
  留下一个未解的钩子，让听众愿意等 30 秒再听你展开。
- **stakes（赌注被抬高）** —— ②-③ 之间：把今天那件大事落到一个
  具体的赌注上——"如果这事走向 A，三年内 X；如果走向 B，Y"
  ——用具体后果而非抽象意义来抬 stakes。
- **turn（视角反转）** —— ③ 内部 / ③-④ 之间：把同一件事用跨域碰
  撞/反方对比重新看一遍，"顺着这条线但换一个坐标系"——让
  听众意识到刚才 ②-③ 的理解只是入口。
- **payoff + 自我披露 + 温度落点（self-disclosure & landing）** ——
  ④ 的收尾：把跨域碰撞的洞察收回到主持人自己的判断上，写一句
  "我之所以这么在意这件事，是因为……"——这是人格锚点，让听众
  带走的不是知识点，而是"这人对这事是真的有感觉的"。整期落在一个
  **有温度的收尾观察**上，而不是一份"我下注：1…2…3…"的清单。

**vault-leads（用户的思考是主线，新闻是镜头）**：

早间档的核心论断应该来自 vault 笔记的某条长期观察/反复印证
的张力，新闻只是放大这个观察的那面镜头。写法上 ①-② 衔接处
点明："这事之所以击中我，是因为我之前一直有个不太说得清的
想法……"——把 vault 的长期观察先于新闻抬出来，再用新闻
去印证/动摇它。这一段不要写得像"参考了我之前一篇笔记"，
要用自然回味的口吻（"这半年我一直在想……"/"前阵子我心里
一直有一个不太说得清的事……"）。

1. **① 开场·自我证伪**（写在最前面）—— Upstream Artifacts 注入了
   「到期旧注结算段」和「最近开放问题段」（含昨晚晚间档当天写入的
   注）。先结算/承接它们：用卞旸第一人称回顾「我上次说 X，证据走向 Y，
   我对/错在哪，更新后的看法是什么」；找不到真实矛盾就跳过一段、直接
   进入 ②。这段是开场不是中后段附录。
2. **② 抛出今天那件大事** —— 从 `brief-X.approved_topics` 挑今天世界
   真正发生的那件大事，用卞旸口吻讲清楚它是什么、发生在哪里、关键事
   实/数字/主体是谁。禁止包装成"AI 行业动态"——它是哪域就是哪域。
3. **③ 我的理解与洞察** —— 给出新闻源给不了的角度：为什么这件事重
   要、它揭示的底层结构是什么、它和过去类似事件的差别在哪。这一段是
   这期播客的价值核心。
4. **④ 关联周边事件 + 温度落点收尾** —— 先把它和其它领域的动向连起
   来，结合 `brief-X.cross_domain_candidates`（本路强制 domain 已在
   Collection 步指定）至少用一条做真实的跨域碰撞，用自然回味的口吻提到
   （"前两天读到个说法"/"最近一直在想"），不报出处、不出现"我的笔记/X
   月 X 日的笔记"这类打卡式溯源；并按 `brief-X.contrarian_source` 做一次
   反方对比：引出它的反方观点，然后正面回应。**然后落在一个有温度的收
   尾观察上收束整期**——把洞察收回到主持人自己的判断（self-disclosure
   人格锚点），自然带出本期至少一个可证伪判断（具体时间窗口/路径/可观
   测信号），但**它是收尾论证的一部分、织在话里，不是一份"我下注：1…
   2…3…"的清单段**。

**可证伪观点：织入，不列单。** 本期必须至少有一个可证伪判断（"我赌
X 在 Y 时间窗内不发生，因为……"），但它**自然长在 ③ 的洞察或 ④ 的
收尾里**，作为论证的一部分说出来，而不是单列一个"我下注的可证伪判断"
格式段。下游会从正文里把它结构化提取进幕后台账（听众无感）。禁止泛泛
的"值得关注"、"持续观察"。

**「上期成绩单」戏剧化结算段（Phase 5, dramatized settlement）**：

① 段开场自我证伪结算不再只是平铺直叙的"上次我说了 X，证据
走向 Y"——把它做成一个戏剧化的小段，**用一句固定的开场语
和一句固定的收束语包住**，让它听起来像一个**有标签的固定
环节**（类似电台的"档案柜时间"），而不是正文里随便一段。
   - 固定开场语：例如"上期成绩单" / "上次的话，回来对一下账"。
   - 固定收束语：例如"这一段到这里，下一件事——"。
   - 中间是真正的 1-2 个 due bet 结算（"上期我说 X，证据走到
     Y，所以我对/错在……"）。
   - 如果这一期没有 due bet，**整段就跳过**——不要为了对齐
     模板而硬写"今天没有要结算的"，那是凑字。

**硬约束**：

- 严禁伪造第一人称证据：没有真实记录支撑的"我亲测了 8%/跑了 200 轮"
  一律不许写——省略优于编造。二手信息必须显式标注"二手"。
- 命名仪式/亲验/反方对比不是每期必填的强制槽位——是"挣来才有"。
- brief 里 `pkos_note` 至少自然融入一次（不报出处、不念"笔记"）；
  `self_past_candidates` 有真实立场矛盾才写自我对话，否则跳过；
  `named_concept_prompt` 真的能命名才命名一段。
- 严格按 `brief.approved_topics` 写稿，不允许写 brief 之外的 topic；
  每个主题用 brief 指定的 `required_angle` 作为切入视角。
- 单旗舰事件值得讲透——长度来自观点、洞察、张力的层次，不是凑字、
  不是堆论据；宁可短也不硬凑、不编造。

**输出**（早间档的写稿步骤）：回复正文必须就是这篇播客文字稿本身，
从 ① 开场·自我证伪到 ④ 关联周边+温度落点收尾，完整、约 7000 字，直接写在回复
里。不写文件、不写过程说明、不写摘要。

### Critique (老黑) — JSON 评分与反方证据

读取上游"写稿X"产出的草稿。按以下 JSON schema 输出 critique（仅
JSON、不要前置或后置说明、不要 code fence）：

```json
{
  "theses": [
    {
      "statement": "<原论断文字>",
      "falsifiability_score": <0..1>,
      "counter_evidence": "<反方证据的简短描述>",
      "response_required": <true|false>
    }
  ],
  "missing_constraints": ["<必须补全的内容描述>"],
  "suggested_revisions": ["<建议修订的具体表述>"]
}
```

要求：每个论断至少给一条反方证据；`missing_constraints` 来自比对
快刀的 4 条编辑哲学标准（可证伪论断 / 反方证据回应 / 亲验或二手标注
/ 可执行尾巴）。

### Polish (快刀青衣) — 4 段结构保留 + D-006

你是快刀青衣。润色对应路的播客草稿，输出完整逐字稿（目标约 7000 字，篇幅不得向下缩水）。
输入是草稿 + critique JSON。润色时按 `critique.missing_constraints` 补
全所有缺失项，并按 `critique.suggested_revisions` 处理每条建议（采纳
或反驳）。

**正文纯净——硬性要求**：逐字稿正文是给听众的成品。下列内容一律
不得出现在正文里：

- 「老黑批判了一通」「critique 指出」这类对润色/评审过程的叙述
- 「编辑注」「润色说明」这类编辑元信息
- 任何提及"草稿""老黑""快刀""达芬奇""钱钟书"工作流角色的句子

逐字稿从卞旸的开场白直接开始，到卞旸的结束语结束。你对 critique 的
处理体现在正文质量本身。

**四段结构保留**：完整保留草稿的四段事件中心结构，不得拆散重排、不
得重新塞回「概览+N 个知识点」式陈列：

1. ① 开场的自我证伪结算/承接段（草稿里有就完整保留，没有就跳过）
2. ② 今天那件大事（按 `brief.approved_topics` 选出的那件）
3. ③ 我的理解与洞察（本期的可证伪判断常织在这里）
4. ④ 关联周边事件（跨域碰撞 + 反方对比）+ 温度落点收尾（self-disclosure
   人格锚点 + 织入的可证伪判断收束整期）
- **可证伪判断织入正文、不单列格式段**：草稿里那句"我赌 X 在 Y 时间窗
  内…"长在 ③ 洞察或 ④ 收尾的论证里——完整保留这句的可证伪强度（具体
  时间窗口/路径/可观测信号），**不得替换为更模糊的措辞**（下游「本期下
  注」从正文蒸馏它进台账），但**不要把它重新拎出来做成一个"我下注"标
  题段**。

**D-006 硬约束**：

- 不得在润色阶段补造第一人称证据
- 不得在润色阶段硬加命名仪式
- 跨域碰撞段、反方对比段、self_past 自我对话段同理：草稿里实际写
  了什么就保留什么，没写的不要补

**输出**：回复正文必须就是这篇完整逐字稿本身，从卞旸开场白第一个字
到结束语最后一个字，一字不少直接写在回复里。

### Score (钱钟书, structured ONLY — no narrative voice)

读取上游三条完整逐字稿候选：终稿-A、终稿-B、终稿-C。`candidate_id`
固定对应：终稿-A→"稿-A"，终稿-B→"稿-B"，终稿-C→"稿-C"。

按下面 4-KPI 细则给每篇逐字稿打分，每项 1-5 分，锚点对标范本《信息
平原》（约 5/5/4/5）：

- **KPI 洞察（洞察密度）**：1=纯新闻复述无穿透观点；3=有明确立场判
  断含可证伪赌注但停在事实层；4=有跨事实的解释性洞察，解释"为什么
  "；5=原创解释框架，让听众重新理解一整类现象。
- **KPI 命名**：1=无命名尝试；3=有命名但抽象无画面感；4=原创命名有
  画面感但复用性一般；5=3-5 字、画面感强、可复用。
- **KPI 跨域（跨域碰撞）**：1=无跨域引用；3=跨域笔记挂到主线但连接
  表面；4=跨域产生真实类比或张力推进论证；5=跨域碰撞本身就是这期洞
  察来源。
- **KPI 思考问句**：1=结尾泛泛收束；3=有总结无开放问题；4=有引导
  思考的问句但偏泛；5=能让听众停下来的问句。

self-past dialectic（与过去的自己辩论段落）不是独立 KPI，出现且真诚
时抬高洞察与思考问句分。

选出最好的一篇（total = 4 项之和，满分 20；比较时先看洞察、再看跨
域破平）。**不要依赖评分 JSON 的 `selected` 字段**（评审 LLM 可能
标错）——选稿规则在 `lib/episode.py select_draft`：按 `scores.total`
最大，再看 `洞察`，再看 `稿-A < 稿-B < 稿-C` 顺序。

严格按 JSON 结构输出，仅 JSON、不要前后说明、不要 code fence：
`{candidates: [{candidate_id, scores: {洞察, 命名, 跨域, 思考问句,
total, self_past_dialectic_present}, selected, editor_notes}, ...]}`。
`candidate_id` 必须严格用 "稿-A"/"稿-B"/"稿-C"，与上述对应关系一致
——平台靠这个 id 把评分与本路 brief 扰动对齐。三个候选都要在
candidates 数组里，`selected` 恰好一个为 true。

**重要：本步骤是纯结构化评分；不要绑定任何叙事/speakAs 人设，不要
以第一人称叙述者身份发言，不要写正文、不要写导言。**

### 定稿 (快刀青衣) — 命名 + 声音统一 + 写定稿

**选稿已由上游代码 `lib.episode.select_draft` 完成——不是你来选。**
注入给你的是：那篇**已选中的终稿**、评分 verdict（含 editor_notes，
供参考）、以及 Character Bible（声音统一参照）。不要再比较分数、不
要再自己选稿、不要依赖 verdict 的 `selected` 字段。

**你的任务——声音统一**：把这篇已选中的终稿定稿成"一个统一主持人
声音"的成品——worldview / obsessions / verbal tics / evolving stances
全部对齐 Character Bible，无论 committee 哪一稿胜出，成品都要像同一
个主持人。保留早间四段结构（可证伪判断织入正文、不单列下注段），遵守 D-006 硬约束（不补造第一人称证据、
不硬加命名仪式、正文不漏编辑元信息）。

**本步骤最终输出——硬性要求**：输出一个 JSON 对象
`{"title": "<给这期起的简洁标题，≤20 字，不含日期和书名号>",
"body": "<声音统一后的完整正文 markdown>"}`。`title` 自己命名（不要
拿"早间播客"这种固定字符串，要根据本期内容起一个能概括主题的简洁标
题；不要带日期、不要用《》书名号）。`body` 放完整正文（一字不少，
包括 markdown 标题、列表、强调等）。**不要只输出元数据**（如
`candidate_id`、分数等）——`body` 字段必须放完整正文 markdown。

### 口播稿 (卞旸) — 纯文本逐字稿

从定稿结果 JSON 对象的 `body` 字段取正文（不要读整个定稿结果对象，
不要读 `title` 字段，把它当作纯文本逐字稿正文使用）。把这份正文改写
成口播稿——同一套事实/论点/数字/引述/可证伪下注/反方证据全保留，
但全程纯文本，无 markdown 符号，无『核心知识点 N』裸编号。

- 硬性禁止：markdown 语法、裸编号（『核心知识点一』『首先』等）、
  书面化词组（『综上所述』『不难看出』）
- 表达节奏：句子 ≤30 字优先；信号词放句首『所以』『但是』；用『我』
  『你』『我们』直接称呼听众
- 开场 60 字内说今天主题 + 钩子；结尾留一句点睛、不要『以上就是今
  天的内容』
- 长度与 body 等长，不删内容；不复述自己说过的话
- 输出格式：纯文本逐字稿。不带 markdown 符号。可以用空行分段

### TTS (周杰伦)

把口播稿文本路径交给 `tts` 技能（personal-os 舰队成员 `tts-toolkit`）的
`synth-auto` 入口（不要自己写 vendor curl；不要硬编码任何脚本绝对路
径）。脚本会做额度预检、vendor 自动选、限流、跨厂商 fallback。输出
纯人声 mp3，发布为最终音频 artifact。
